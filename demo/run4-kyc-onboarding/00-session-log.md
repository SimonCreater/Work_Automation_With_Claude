# run4 세션 로그 — KYC 온보딩 심사 자동화 도구

> 이 로그는 하네스가 R1(에이전트 선질문) · R2(단계별 승인 게이트) · R3(기계적 검증)을
> 실제로 수행했음을 보이는 증적이다. 본 run은 사용자가 모호한 아이디어("금융 업무 자동화
> 툴")에서 출발해 에이전트와의 인터뷰로 대상을 좁힌 뒤, S1 승인 후 나머지 단계를 위임받아
> 자동 진행한 세션이다. 사용한 하네스(`templates/`, `schemas/`, `scripts/validate.py`, `.claude/`)는 무수정이다 — R4.

## 아이디어 (한 문단, 사용자 입력)
사용자는 처음 financial-datasets MCP 서버(재무제표·주가·뉴스 데이터 공급)와
anthropics/financial-services(금융 업무 자동화 에이전트 모음)를 참고로 가져왔고,
"금융 쪽에서 업무를 자동화하는 툴을 만들고 싶다"는 모호한 의도를 밝혔다. 에이전트가
범위를 좁히는 인터뷰를 통해 **KYC/온보딩 심사 자동화**로 대상을 확정했다.

## S0. 기동 및 범위 좁히기 (R1)
빈 양식 대신 단계별 질문을 먼저 던지는 흐름으로 진입(R1). "금융 업무 자동화"는 단일
시스템으로 설계 불가할 만큼 넓으므로, 후보 4종(KYC 심사 / GL 정산·월마감 / 어닝 노트 /
포트폴리오 리밸런싱)을 제시하고 사용자가 **KYC/온보딩 심사 자동화**를 선택.

## S1. 시스템 요청서
에이전트가 후원자/비즈니스 니즈/제약/범위(특히 Out of Scope) 4문항을 먼저 질문(R1).
사용자가 제안된 기본값을 승인. → `01-system-request.md` 생성 → **검증 통과** → **[승인 게이트]** 승인. (R2)

## (위임) S2~S6 자동 진행
S1 승인 시점에 사용자가 "이제 묻지 말고 끝까지 진행하라"고 위임. 이후 단계는 합리적
기본값으로 에이전트가 작성하되, 각 단계마다 R3 기계 검증을 통과시키는 절차는 그대로 유지.

## S2. 요구사항 정의
기능 요구(FR-1~6)와 비기능 요구(성능·보안·규제·신뢰성·사용성 NFR-1~5)를 검증 가능한 서술로 작성.
→ `02-requirements.md` 생성 → **검증 통과**(FR≥2, NFR≥1).

## S3. 유스케이스 2건
UC-1(서류 자동 심사·케이스 생성), UC-2(케이스 검토·결정). 각각 정상 흐름 + 예외 흐름
(추출 실패/리스트 만료/고유사도 매칭/근거 불충분/고위험 2차 승인) 포함.
→ `03-use-cases.md` 생성 → **검증 통과**(예외 흐름 2건 이상 규칙 충족).

## S4. 구조 다이어그램 (DFD)
컨텍스트 + Level 0(프로세스 1~4, 데이터 스토어 D1~D4), Mermaid flowchart.
→ `04-structure-dfd.md` 생성 → **검증 통과**(mermaid 블록 2개, flowchart).

## S5. ERD
CUSTOMER_PROFILE / ONBOARDING_CASE / MATCH_CANDIDATE / WATCHLIST_ENTRY / REVIEWER / AUDIT_LOG.
DFD의 D1~D4와 대응(balance) 명시.
→ `05-erd.md` 생성 → **검증 통과**(PK, 카디널리티 표기).

## S6. ADR 3건
ADR-1(최종 결정은 사람이 — 1차 심사만 자동화) / ADR-2("블랙박스 ML 스코어링은 도입하지
않기로 한다" — 하지 않기로 한 결정) / ADR-3("자체 워치리스트를 구축·관리하지 않기로 한다").
→ `06-adr.md` 생성 → **검증 통과**("하지 않기로" 규칙 충족).

## 결론
S1~S6 산출물 6종이 모두 R3 검증을 통과(통과 6 / 실패 0)했고, S1은 사용자 승인 게이트를,
이후 단계는 사용자 위임 하에 동일한 기계 검증을 거쳤다. 하네스는 무수정으로 재사용되었다(R4).
