# 유스케이스 (Use Cases)

---

## UC-1: PT 예약 생성

- **ID**: UC-1
- **Priority**: High
- **Primary Actor**: 회원
- **Trigger**: 회원이 트레이너와 PT 세션을 잡고 싶어 예약 화면에 들어온다
- **Trigger Type**: External
- **Description**: 회원이 트레이너와 가능 시간대를 선택해 PT 예약을 생성한다.

### Preconditions (사전 조건)
- 회원이 로그인되어 있다
- 트레이너가 예약 가능 시간대를 1개 이상 등록해 두었다

### Normal Flow (정상 흐름)
1. 회원이 트레이너를 선택한다
2. 시스템이 해당 트레이너의 예약 가능 시간대를 표시한다
3. 회원이 원하는 시간대를 선택한다
4. 시스템이 해당 시간대의 중복 여부를 확인한다
5. 시스템이 예약을 확정하고 회원과 트레이너에게 확인을 표시한다

### Alternate / Exception Flow (대안·예외 흐름)
- 2a. 가능 시간대가 하나도 없으면 시스템이 "예약 가능 시간 없음"을 안내하고 다른 트레이너 선택을 제안한다
- 4a. 선택 직전 다른 회원이 같은 시간대를 먼저 확정했다면 시스템이 충돌을 알리고 시간대 목록을 갱신한다

### Postconditions (사후 조건)
- 중복 없는 예약 1건이 확정 상태로 저장된다

## UC-2: 예약 리마인더 발송

- **ID**: UC-2
- **Priority**: Medium
- **Primary Actor**: 회원
- **Trigger**: 예약 시작 시점 기준 정해진 시간 전이 되면 발송 작업이 실행된다
- **Trigger Type**: Temporal
- **Description**: 시스템이 다가오는 예약에 대해 회원에게 리마인더를 보내 노쇼를 줄인다.

### Preconditions (사전 조건)
- 확정 상태의 예약이 존재한다

### Normal Flow (정상 흐름)
1. 정해진 시각에 시스템이 다가오는 확정 예약을 조회한다
2. 시스템이 각 예약의 회원 연락처를 확인한다
3. 시스템이 리마인더 메시지를 구성한다
4. 시스템이 회원에게 리마인더를 발송한다
5. 시스템이 발송 완료를 예약에 기록한다

### Alternate / Exception Flow (대안·예외 흐름)
- 1a. 대상 기간에 다가오는 예약이 없으면 시스템은 아무 것도 발송하지 않고 종료한다
- 4a. 발송이 실패하면 시스템이 일정 횟수 재시도하고, 계속 실패하면 점장에게 실패 목록을 보고한다

### Postconditions (사후 조건)
- 다가오는 예약의 회원이 리마인더를 수신하고, 발송 결과가 기록된다
