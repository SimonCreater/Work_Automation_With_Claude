# run2 회고 — 다른 도메인 무수정 재적용

대상 도메인: 동네 헬스장 PT 예약 관리 (금융 도메인인 run1과 의도적으로 멀리 잡음).
하네스(`templates/`, `schemas/`, `scripts/validate.py`, `.claude/`)는 **한 글자도 수정하지 않고** 재실행했다.

1. 잘 된 점: 산출물 6종이 모두 검증을 통과했고, 도메인 용어("종목/공시" → "회원/예약")가 전부 인터뷰 값으로만 바뀌어 R4(재사용성)가 실제로 성립했다.
2. 잘 된 점: temporal 트리거 유스케이스(run1=공시 모니터링, run2=리마인더)가 두 도메인 모두 자연스럽게 나왔다 — 단계 순서가 이론 흐름을 따르기 때문이다.
3. 한계: 검증 훅은 **형식만** 본다. run2 ERD에서 BOOKING:ATTENDANCE를 1:1로 뒀는데, 이게 도메인적으로 맞는지는 기계가 못 보고 사람 승인에 의존했다. 스키마 설계상 내용 타당성은 검증 범위 밖이다.
4. 한계: DFD-ERD 정합(balance, ch05/06)을 훅이 강제하지 못한다. D2(예약 내역)와 BOOKING 엔티티의 대응은 사람이 눈으로 확인했을 뿐, "데이터 스토어=엔티티" 자동 대조 규칙은 없다.
5. 한계: 인터뷰 질문이 도메인 비종속이라, "동시 예약 충돌" 같은 도메인 특화 쟁점은 에이전트가 먼저 묻지 못하고 사용자가 꺼내야 나왔다 — R4를 지키려고 질문을 일반화한 설계의 대가다.
6. 한계: Mermaid 문법 오류는 검증되지 않는다. `validate.py`는 ```mermaid 블록 존재와 flowchart/erDiagram 키워드만 보고, 실제 렌더링 가능 여부는 GitHub에 올려야 안다.
7. 한계: 산출물 순서가 S1~S6으로 고정이라, 예약 도구처럼 비교적 단순한 아이디어엔 ERD까지 6단계가 다소 무겁게 느껴졌다(과제 요건이라 유지).
8. 한계: `no_placeholders` 규칙이 단순 문자열(`{{`) 검사라, 본문에 우연히 중괄호가 필요한 도메인(코드 예시 등)에서는 오탐 가능성이 있다.
9. 설계적 교훈: "검증은 형식, 승인은 사람"이라는 경계가 한계의 원인이자 동시에 안전장치였다 — 기계가 통과시켜도 사람이 내용 책임을 지는 구조가 오히려 신뢰를 줬다.
10. 개선 방향: 향후 DFD 데이터 스토어와 ERD 엔티티명을 교차 대조하는 balance 검증 규칙, 도메인 키워드에 따라 선택적 질문을 더하는 hook을 추가하면 R4를 깨지 않으면서 깊이를 높일 수 있다.
