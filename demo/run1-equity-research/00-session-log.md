# 협의 세션 로그 — run1 (종목 리서치 어시스턴트)

> 이 로그는 하네스가 R1(에이전트 선질문) · R2(단계별 승인 게이트) · R3(기계적 검증)을
> 실제로 어떻게 수행했는지 보여주는 증적이다.

---

## 시작 — 아이디어 한 문단 (사용자 입력)

> "종목명만 넣으면 공시·실적·뉴스를 요약해 리서치 메모 초안을 만들어주는 도구를 만들고 싶어"

에이전트는 빈 양식을 주지 않고 `spec-pipeline` 스킬로 **먼저 질문을 시작**했다. (R1)

---

## S1. 시스템 요청서

**[에이전트 질문]**
1. 이 도구를 가장 필요로 하는 사람은 누구인가요? (후원자)
2. 지금은 이 일을 어떻게 하고 있고, 무엇이 가장 불편한가요?
3. 잘 되면 무엇이 좋아지나요? 가능하면 수치로요.
4. 반드시 지켜야 할 제약(법규·데이터 출처 등)이 있나요?

**[사용자 답 요약]** 증권사 리서치센터 애널리스트 / 자료 수집에 종목당 3시간 / 30분으로 줄이고 싶다 / 공식 공시만, 자동매매 금지.

→ `01-system-request.md` 생성 → **검증 통과** → **[승인 게이트]** "이대로 확정하고 요구사항 단계로 갈까요?" → 사용자: **승인**. (R2)

---

## S2. 요구사항 정의

**[에이전트 질문]** 꼭 되어야 하는 기능 / 속도·신뢰성·규제 기준을 물음.
**[사용자 답]** 종목 식별, 공시·실적·뉴스 수집·요약, 메모 초안 생성 / 3분 이내 / 모든 수치에 출처.

→ `02-requirements.md` 생성 → **검증 통과** → **[승인 게이트]** 승인.

---

## S3. 유스케이스  ← R3 검증 실패가 발생한 단계

**[에이전트 질문]** 주요 시나리오와 시작 이벤트, 잘못될 수 있는 경우를 물음.

**[1차 생성]** UC-1(메모 생성), UC-2(공시 모니터링)을 작성했으나 **정상 흐름만** 기재.

**[R3 검증 실패]** PostToolUse 훅이 다음을 보고:
```
[검증 실패] 03-use-cases.md (use-cases) — 1건 위반:
  - 각 유스케이스에 대안/예외 흐름이 반드시 필요합니다 (정상 흐름만 있으면 안 됨). (현재 0건)
```
**[자가 수정]** 에이전트가 동명 기업 처리·데이터 출처 장애·뉴스 0건 등 예외 흐름을 추가해 다시 저장 → **검증 통과**.

→ **[승인 게이트]** 승인. (정상 흐름만으로는 게이트에 도달조차 못 했다 — 검증이 사람보다 먼저 걸러냄)

---

## S4. 구조 다이어그램 (DFD)

**[에이전트 질문]** 어떤 외부 시스템과 주고받는지, 무엇을 저장하는지 물음.
→ `04-structure-dfd.md` 생성(컨텍스트 + Level 0, Mermaid) → **검증 통과** → 승인.

---

## S5. ERD

**[에이전트 질문]** 저장 대상(종목·공시·뉴스·메모)과 관계를 물음.
→ `05-erd.md` 생성 → **검증 통과** → 승인. (DFD의 데이터 스토어 D1~D3와 엔티티가 대응됨)

---

## S6. ADR

협의 전반에서 포착해 둔 결정을 정리:
- ADR-1: 공식 공시 출처만 사용
- ADR-2: 목표주가 자동 산정은 **하지 않기로** 결정 (규제·오도 리스크)

→ `06-adr.md` 생성 → **검증 통과**("하지 않기로 한 결정" 포함 규칙 충족) → 승인.
