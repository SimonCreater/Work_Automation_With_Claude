# spec-pipeline — 협의→산출물 파이프라인 (자동화①)

이 스킬은 `.claude/CLAUDE.md`의 운영지침을 실제로 굴리는 파이프라인이다.
아이디어 한 문단을 받아 6개 산출물을 순서대로 생성한다. 각 단계는 **생성 → 기계 검증 → 사용자 승인**의 3박자를 반드시 지킨다.

## 시작

1. 사용자에게 아이디어 한 문단을 받는다. (이미 받았다면 생략)
2. run 이름을 정한다(예: `run1-<도메인키워드>`). 산출물은 `demo/<run>/`에 저장한다.
3. S1부터 시작한다.

## 단계 루프 (S1 → S6)

각 단계마다 아래를 그대로 수행한다:

1. **인터뷰**: 해당 단계의 질문을 2~4개씩, 좁고 구체적으로 던진다. (먼저 묻는다 — R1)
   모르는 값은 기본값을 제안하고 확인받는다. 결정·거부한 대안은 ADR 후보로 메모해 둔다.
2. **생성**: `templates/NN-*.md`를 복제해 응답으로 채운 뒤 `demo/<run>/NN-*.md`로 Write 한다.
   placeholder(`{{ }}`)를 하나도 남기지 않는다.
3. **검증**: Write 직후 PostToolUse 훅(`scripts/validate.py`)이 자동 실행된다.
   stderr에 위반이 보고되면 **그 항목만 고쳐 다시 Write** 한다. 통과할 때까지 반복.
4. **승인 게이트(R2)**: 검증을 통과하면 산출물 핵심을 3~5줄로 요약해 보여주고,
   "이대로 확정하고 다음 단계(<다음 산출물>)로 갈까요? (승인 / 수정)"라고 **반드시 묻는다.**
   사용자가 승인하기 전에는 다음 단계로 넘어가지 않는다. 수정 요청이면 같은 단계에서 반영 후 다시 게이트.

| 단계 | 산출물 파일 | 인터뷰 핵심 |
|---|---|---|
| S1 | 01-system-request.md | 후원자 / 비즈니스 니즈 / 기대 가치 / 제약 / 범위 |
| S2 | 02-requirements.md | 기능 요구(FR) / 비기능 요구(NFR: 성능·보안·규제 등) |
| S3 | 03-use-cases.md | 이벤트·액터·트리거 / 정상 흐름 / **예외 흐름** (2건 이상) |
| S4 | 04-structure-dfd.md | 프로세스(동사구) / 데이터 흐름 / 외부 엔티티 / 데이터 스토어 |
| S5 | 05-erd.md | 엔티티 / 속성·PK / 관계·카디널리티 (DFD 데이터 스토어와 정합) |
| S6 | 06-adr.md | 협의 중 포착한 결정 2건+ (1건은 "하지 않기로 한 결정") |

## 종료

S6까지 모두 승인되면, `demo/<run>/`에 6종 산출물이 모인다.
세션 로그(질문·답변·검증·승인 흐름)를 함께 보존하면 평가 증적이 된다.

## 재사용성(R4)

도메인 용어를 절대 하드코딩하지 않는다. 모든 값은 인터뷰에서만 주입한다.
같은 스킬을 다른 도메인에서 무수정으로 다시 실행할 수 있어야 한다.
